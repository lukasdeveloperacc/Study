# MVP 설계 문서: 지역여행계획 플래너

## 1. 문제 정의

### 1.1 핵심 문제
- **계획의 귀찮음**: 여행 계획을 세우는 과정 자체가 번거롭고 시간이 많이 소요됨
- **정보 파편화**: 여행 정보가 인스타그램, 블로그, 검색 등 여러 곳에 흩어져 있어 일일이 찾아다녀야 함
- **의사결정 피로**: 너무 많은 선택지와 정보로 인해 무엇을 선택해야 할지 어려움
- **동선 최적화 실패**: 장소는 찾았지만 효율적인 동선을 짜기 어려움

### 1.2 타겟 고객
**주요 타겟**
- 연인/커플 (높은 여행 빈도)
- 여행 애호가 (자주 국내 여행을 다니는 사람)
- 20-30대 직장인 (주말 여행, 단기 여행)

**고객 특성**
- 여행은 가고 싶지만 계획 세우는 시간이 아까운 사람
- 빠르게 여행 계획 초안을 받고 싶은 사람
- 핫플레이스/맛집 정보를 한 곳에서 얻고 싶은 사람

### 1.3 핵심 가설
**가설 1**: 사용자들은 여행 계획을 세우는 시간을 줄이기 위해 AI 자동 생성 서비스를 사용할 것이다
- 검증 방법: 첫 방문 후 계획 생성 완료율 측정 (목표: 70% 이상)

**가설 2**: 자동 생성된 계획도 일부 수정 기능은 필요하다
- 검증 방법: 생성 후 수정 기능 사용률 측정 (예상: 40-60%)

**가설 3**: PDF 출력 기능은 실제 여행 시 활용도가 높다
- 검증 방법: PDF 다운로드율 측정 (목표: 50% 이상)

## 2. 핵심 기능 정의

### 2.1 사용자 여정 (User Journey)

```
1. [접속] 웹/앱 접속
   ↓
2. [입력] 여행지 입력 (예: "강릉")
   ↓
3. [선호도] 여행 선호도 입력
   - 여행 기간 (1박2일, 2박3일 등)
   - 여행 인원 (혼자, 커플, 가족, 친구)
   - 예산 범위 (경제적, 보통, 여유롭게)
   - 여행 스타일 (휴양, 액티비티, 맛집투어, 관광, 자연)
   - 선호 카테고리 (바다/산, 카페/맛집, 문화/역사)
   - 이동 수단 (자차, 대중교통)
   ↓
4. [생성] AI가 자동으로 여행 계획 생성 (10-30초)
   ↓
5. [확인] 생성된 계획 확인
   - 일차별 일정
   - 시간대별 방문 장소
   - 숙소 추천
   - 장소 설명 및 사진
   - 이동 시간 및 동선
   - 예상 비용
   ↓
6. [선택] 사용자 의사결정
   ├─ 만족 → [PDF 출력] → 여행 준비 완료
   └─ 불만족 → [수정] → 다시 확인
      - 장소 삭제/추가
      - 순서 변경
      - 시간 조정
      - 전체 재생성
```

### 2.2 필수 기능 (MVP 포함)

#### 1순위: 핵심 기능
- **여행지 입력**: 간단한 텍스트 입력으로 목적지 지정
- **선호도 설문**: 6가지 항목 (기간, 인원, 예산, 스타일, 카테고리, 이동수단)
- **AI 자동 일정 생성**: LangGraph 기반 여행 계획 자동 생성
  - 숙소 추천
  - 시간대별 방문 장소 (오전/점심/오후/저녁)
  - 장소 간 이동 시간 계산
  - 예상 비용 산출
  - 장소 설명 및 이미지
- **간단한 수정 기능**:
  - 장소 삭제
  - 장소 추가 (AI가 대안 제시)
  - 순서 변경 (드래그앤드롭)
  - 전체 재생성
- **PDF 출력**: 생성된 일정을 PDF로 다운로드

#### 2순위: 향후 추가 기능 (MVP 이후)
- 일정 저장 및 마이페이지
- 일정 공유 기능 (링크, SNS)
- 실시간 날씨/교통 정보
- 사용자 리뷰 및 평점
- 실제 예약 연동 (숙소, 맛집 예약)
- 커뮤니티 기능 (일정 공유, 후기)

### 2.3 제외 기능 (MVP에서 제외 이유)

| 기능 | 제외 이유 | 추가 시기 |
|------|-----------|-----------|
| 회원가입/로그인 | 진입장벽 낮추기, MVP는 빠른 검증 우선 | v1.1 |
| 일정 저장/관리 | 로그인 없이 즉시 사용 가능하게 | v1.1 |
| 실시간 날씨/교통 | 외부 API 의존도 증가, 복잡도 상승 | v1.2 |
| 예약 연동 | 제휴 필요, 개발 기간 증가 | v2.0 |
| 커뮤니티 | 초기 사용자 부족 시 효과 미미 | v2.0 |
| 다국어 지원 | 국내 여행 집중 | v2.0 |
| 모바일 앱 | 웹으로 먼저 검증 후 앱 개발 | v1.5 |

### 2.4 기능 우선순위

**P0 (필수, MVP 없이 불가능)**
1. 여행지 입력 UI
2. 선호도 설문 폼
3. AI 일정 생성 엔진
4. 일정 조회 화면
5. PDF 출력 기능

**P1 (중요, MVP 핵심 가치)**
1. 장소 삭제 기능
2. 장소 순서 변경
3. 전체 재생성 기능
4. 숙소 추천
5. 예상 비용 표시

**P2 (좋으면 좋은, 시간 있으면 추가)**
1. 장소 추가 기능 (AI 대안 제시)
2. 세부 시간 조정
3. 지도 연동 (카카오맵/네이버맵 링크)
4. 로딩 중 재미있는 메시지

## 3. MVP 제작 계획

### 3.1 기술 스택

#### Frontend
- **React 18**: 빠른 개발, 풍부한 라이브러리
- **Vite**: 빠른 개발 환경, HMR
- **TailwindCSS**: 빠른 스타일링, 템플릿 활용 가능
- **React DnD**: 드래그앤드롭 (일정 순서 변경)
- **React-PDF**: PDF 생성
- **Axios**: API 통신

**추천 UI 템플릿**
- Flowbite (무료, Tailwind 기반)
- DaisyUI (무료, Tailwind 기반)

#### Backend
- **FastAPI**: 빠른 개발, 자동 문서화, 비동기 지원
- **Python 3.11+**: 최신 성능 개선
- **PostgreSQL**: 관계형 데이터 저장 (장소, 리뷰 등)
- **SQLAlchemy 2.0**: ORM
- **Pydantic**: 데이터 검증

#### AI/LLM
- **LangGraph**: 복잡한 여행 계획 워크플로우 구성
- **OpenAI GPT-4o-mini**: 비용 효율적, 빠른 응답
  - 예상 비용: 요청당 약 50-100원 (1000토큰 기준)
  - 월 100명 사용 가정: 5,000-10,000원
- **LangChain**: 프롬프트 관리, 메모리
- **Vector DB (Pinecone 무료 티어 또는 PostgreSQL pgvector)**: 장소 임베딩 검색

#### 데이터 수집
- **크롤링 → DB 저장 방식** (강력 추천)
  - Beautiful Soup / Selenium: 크롤링
  - 소스: 네이버 블로그, 인스타그램 위치 태그, 구글 맵 리뷰
  - 초기 지원 지역: **인기 여행지 5-10곳**
    - 강릉, 부산, 제주, 경주, 전주, 여수, 속초, 춘천, 가평, 대구
  - 크롤링 주기: 주 1회 (최신 정보 유지)

**크롤링 후 DB화가 나은 이유:**
- ✅ 빠른 응답 속도 (DB 쿼리 < 1초 vs 실시간 크롤링 10-30초)
- ✅ 안정성 (외부 사이트 차단/변경에 독립적)
- ✅ 데이터 품질 관리 (잘못된 정보 필터링 가능)
- ✅ 비용 절감 (반복 크롤링 방지)

#### 인프라
- **Vercel** (프론트): 무료, 자동 배포, CDN
- **Railway / Render** (백엔드): 무료 티어 충분
- **Supabase** (DB): PostgreSQL 무료 티어 500MB
- **도메인**: Vercel 기본 도메인 또는 가비아 연간 1-2만원

**예상 월 비용**: 1-2만원 (OpenAI API + 도메인)

### 3.2 팀 구성 및 역할

**1인 개발자 (Full-stack)**
- Frontend 개발: React, UI/UX
- Backend 개발: FastAPI, DB 설계
- AI 개발: LangGraph, 프롬프트 엔지니어링
- 데이터 수집: 크롤링 스크립트 작성
- 배포 및 운영: DevOps

**외주/아웃소싱 검토**
- ❌ 개발: 1인 개발로 진행
- ⚠️ 디자인: 템플릿 활용 (선택적으로 로고/아이콘만 외주, 10-30만원)
- ⚠️ 데이터 수집: 초기 크롤링 데이터 구매 고려 (업체: 크몽, 탈잉, 10-50만원)
  - 장소 데이터 (이름, 주소, 카테고리, 이미지, 설명)
  - 여행지당 100-200개 장소

### 3.3 개발 일정 (4주)

#### Week 1: 기반 구축 및 데이터 준비
**Day 1-2: 프로젝트 셋업**
- React + Vite 프로젝트 생성
- FastAPI 프로젝트 생성
- PostgreSQL 설정 (Supabase)
- Git 저장소 생성
- 기본 폴더 구조 설계

**Day 3-5: 데이터베이스 설계 및 크롤링**
- DB 스키마 설계
  - places (장소), regions (지역), categories (카테고리)
- 크롤링 스크립트 작성
  - 네이버 블로그, 인스타그램 위치 태그
- 초기 데이터 수집 (강릉, 부산, 제주 우선)
- DB 적재

**Day 6-7: AI 엔진 프로토타입**
- LangGraph 워크플로우 설계
  - 입력 파싱 → 장소 선택 → 동선 최적화 → 일정 생성
- 간단한 프롬프트로 일정 생성 테스트
- OpenAI API 연동

#### Week 2: 핵심 기능 개발
**Day 8-10: Backend API 개발**
- POST /api/plan/generate: 일정 생성 API
  - 입력: 여행지, 선호도
  - 출력: JSON 형태 일정
- GET /api/places: 장소 목록 조회
- GET /api/regions: 지원 지역 목록
- FastAPI 문서 자동 생성 확인

**Day 11-14: Frontend UI 개발**
- 홈페이지: 여행지 입력 + 선호도 설문
- 로딩 화면: AI 생성 중 표시
- 일정 조회 화면: 생성된 일정 표시
  - 일차별 탭
  - 시간대별 카드
  - 장소 정보 (이름, 사진, 설명, 이동시간)
- 기본 반응형 디자인 (모바일/데스크톱)

#### Week 3: 수정 기능 및 PDF
**Day 15-17: 수정 기능 구현**
- 장소 삭제 버튼
- 순서 변경 (React DnD)
- 전체 재생성 버튼
- 수정 API 연동
  - PUT /api/plan/update

**Day 18-21: PDF 출력 기능**
- React-PDF로 PDF 템플릿 디자인
- 다운로드 버튼 구현
- PDF 내용: 전체 일정 + 지도 링크

#### Week 4: 최적화 및 배포
**Day 22-24: 테스트 및 버그 수정**
- 전체 플로우 테스트
  - 여행지 입력 → 생성 → 수정 → PDF
- 엣지 케이스 처리
  - 지원하지 않는 지역 입력
  - AI 생성 실패 시 재시도
- 성능 최적화
  - API 응답 시간 < 10초
  - 페이지 로딩 < 2초

**Day 25-26: 배포**
- Vercel에 프론트엔드 배포
- Railway/Render에 백엔드 배포
- 환경 변수 설정 (API Key 등)
- HTTPS 설정
- 도메인 연결 (선택)

**Day 27-28: 피드백 수집 준비**
- Google Analytics 연동
- 간단한 피드백 폼 추가
- 사용 가이드 작성

### 3.5 성공 지표 (KPI)

#### 제품 지표
1. **완료율**: 첫 방문 후 일정 생성 완료 (목표: 70%)

3. **PDF 다운로드율**: 생성된 일정 중 PDF 다운로드 (목표: 50%)

3. **페이지 로딩 속도**: 2초 이내

3. **만족도**: 설문 조사 4.0/5.0 이상

### 4.2 검증 계획

#### 가설 1: 사용자들은 자동 생성 서비스를 사용할 것이다
**검증 방법**
- 완료율 측정: (일정 생성 완료 수 / 첫 방문 수) × 100
- 목표: 70% 이상
- 측정 도구: Google Analytics 이벤트 트래킹

**실패 시 대응**
- 완료율 < 50%: 선호도 설문이 너무 길거나 복잡 → 간소화
- 생성 시간 > 30초: 성능 최적화 또는 로딩 UX 개선

#### 가설 2: 수정 기능은 필요하다
**검증 방법**
- 수정률 측정: (수정한 사용자 수 / 생성한 사용자 수) × 100
- 목표: 40-60%

**결과 해석**
- 수정률 < 20%: 초기 생성 품질이 매우 높음 → 수정 기능 최소화 가능
- 수정률 > 80%: 초기 생성 품질 낮음 → AI 프롬프트 개선 필요

#### 가설 3: PDF 출력은 활용도가 높다
**검증 방법**
- PDF 다운로드율: (PDF 다운로드 수 / 일정 생성 수) × 100
- 목표: 50% 이상

**실패 시 대응**
- 다운로드율 < 30%: 사용자들이 스마트폰으로 직접 확인 선호 → 모바일 최적화 강화
